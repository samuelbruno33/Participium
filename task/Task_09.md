# Task 9 - Frontend Selenium UI Acceptance Testing

## Current testing phase

The goal of this task is to prepare a browser-level acceptance test suite for the implemented Participium frontend.

This task is different from the backend pytest, coverage, and Postman tasks. Its purpose is to exercise the running web application from the user's perspective through a real browser, while the frontend communicates with the backend through the published REST API.

## Deliverables to produce

Complete the following deliverable:

- the Selenium test suite under 
**`../src/frontend/tests/selenium/`**

Do not produce a specific documentation deliverable for this task.

## General guidance

### Treat the suite as browser acceptance testing

The Selenium tests must interact with the running frontend through the browser.

Do not call React components directly. Do not call backend Python functions directly. Do not access the database directly from Selenium tests. Do not fake authentication state by manually injecting cookies or local storage values unless the application itself exposes that mechanism.

Use the UI and the public backend behavior as a user would.

### Use stable HTML IDs

The frontend exposes stable HTML `id` attributes for application-owned elements.

Use ID-based selectors in Selenium, for example:

```python
driver.find_element(By.ID, "login-submit")
```

Do not use `data-testid`, `data-test`, or other `data-*` testing attributes as selector strategy.

For data-generated elements, use the documented ID prefix conventions from `../src/frontend/src/domIds.ts`, for example:

- `public-report-row-<id>`
- `assigned-report-status-<id>`
- `admin-user-save-<id>`
- `notification-read-<id>`

The test code should not depend on CSS class names, text layout, table column position, or third-party library internals when a stable HTML ID is available.

### Keep tests repeatable

The suite must be executable more than once against a local development setup.

Use seeded demo accounts where appropriate:

- `citizen@example.com` / `Citizen123!`
- `operator@example.com` / `Operator123!`
- `admin@example.com` / `Admin123!`

When a test creates new data, use unique values to avoid collisions across repeated runs.

Do not rely on database row order unless the UI explicitly guarantees it. Prefer locating a created resource by a unique title, email, or another value generated by the test.

## 1) Local Selenium Environment

The Python virtual environment is a local execution setup for running the frontend browser tests.

Install dependencies from:

- **`../src/frontend/tests/selenium/requirements.txt`**

### Windows setup

Open a terminal in `../src/frontend`:

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
python -m pip install -r tests\selenium\requirements.txt
```

### Unix / macOS setup

Open a terminal in `../src/frontend`:

```bash
python3 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
python -m pip install -r tests/selenium/requirements.txt
```

### Browser requirements

Use Chrome or Chromium for the default local execution.

## 3) Required Test Coverage

The Selenium suite should provide **100% coverage of the official use cases** defined in `../doc/OfficialDocumentation.md`.

Every use case from UC-01 to UC-16 should be covered by at least one browser test. Test names, docstrings, or helper-level comments should make the covered UC IDs easy to trace.


### Coverage expectations

The tests should:

- use explicit waits for page state, navigation, and asynchronous data loading;
- make assertions on visible behavior and relevant HTML state;
- avoid brittle sleeps;
- use stable HTML IDs before considering any other selector;
- keep test data deterministic and easy to understand;
- fail with useful messages when an expected page element or workflow is missing.

---

## 4) Quality Criteria

### Acceptance value

The suite should read as a coherent browser acceptance campaign, not as isolated clicks.

It should demonstrate that the frontend supports the documented workflows and exposes enough stable HTML structure to make those workflows testable through Selenium.

### Maintainability

The suite should be maintainable:

- clear file organization;
- meaningful test names;
- shared setup extracted into fixtures or page objects where useful;
- no duplication;
- explicit credentials and environment configuration;
- deterministic test data generation.

### Independence from implementation internals

The tests must validate externally observable behavior.

They should not depend on React component structure, internal state variables, database IDs that are not obtained through the UI, backend implementation details, or local database access.

### Executability

The submitted test suite should pass with the default local seed data or clearly create the data it needs through the application UI and public application behavior.
