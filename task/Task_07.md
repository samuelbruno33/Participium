# Task 7 - Implementation Coverage Testing

## Current testing phase

The goal is to extend and organize the automated test suite in order to obtain the highest feasible code coverage on the current implementation.

Unlike the previous black-box and white-box tasks, this task does not require a dedicated analysis report. The authoritative evidence for the result is the coverage report generated by the test framework.

## Deliverables to produce

Complete the following deliverables:

- the automated tests under 
  - `.. /src/backend/tests/unit`
  - `.. /src/backend/tests/integration`
  - `.. /src/backend/tests/e2e`

You may create new subdirectories if they help organize the test campaign.

Do not produce a specific documentation deliverable for this task.

The coverage result must be obtained by running the project test framework and its coverage tool. The generated coverage report is the reference used to evaluate the work.

## Baseline scope reference

Work on the material provided under:

- **`../src/backend/`**

The target is the implemented backend code, including application logic, services, routes, validation paths, helpers, and relevant error handling.

## General guidance

### Maximize meaningful coverage

The test campaign should aim for the maximum feasible coverage of the implementation.

The suite should cover:

- successful execution paths,
- invalid input paths,
- exception paths,
- authorization and permission checks,
- boundary cases,
- relevant state transitions,
- interactions with repositories, services, or external collaborators,
- API-level behavior where meaningful.

High coverage must not be obtained through superficial tests that execute code without checking behavior. Each test must include meaningful assertions on returned values, raised exceptions, state changes, HTTP responses, database effects, or collaborator interactions.

### Organize the test campaign

The test suite must be organized as a maintainable test campaign, not as a collection of unrelated cases.

Tests should be grouped by functional area, module, service, route, or behavior. Use fixtures, helpers, factories, and parametrization where they improve clarity and reduce repetition.

The structure should make it easy to understand:

- what part of the implementation is being tested,
- which behavior each test verifies,
- which setup is shared,
- which cases are intentionally different.

### Follow good programming practices

The test code must follow normal software engineering quality expectations.

The suite should be:

- modular,
- readable,
- maintainable,
- non-redundant,
- free from unnecessary duplication,
- explicit about relevant setup and assertions,
- consistent with the style already used in the project.

Avoid large copy-pasted setup blocks. Prefer reusable fixtures or helper functions when the same arrangement is needed by multiple tests.

### Isolate external dependencies

When tests target internal logic, mock collaborators that are outside the behavior under test, such as external APIs, mail services, storage services, notification gateways, or slow infrastructure.

When tests target integration behavior, keep the setup controlled and reproducible. Do not depend on external services unless the project already provides a reliable local test configuration for them.

### Keep tests deterministic

The test suite must be repeatable.

Avoid tests that depend on:

- execution order,
- real time without control,
- random values without fixed seeds,
- existing local data,
- external network availability,
- manually configured state.

Use explicit fixtures and controlled test data.

### Technical examples

Two example files are provided to show the mechanics of using an in-memory SQLite datasource:

- **`../src/backend/tests/integration/test_category_repository_example.py`**
- **`../src/backend/tests/e2e/test_category_seed_example.py`**

These files are intentionally simple and direct. They show how to:

- configure the test database URL with `monkeypatch`,
- open and close a database connection manually in an integration test,
- create the schema explicitly when the Flask application is not started,
- use the Flask test client in an end-to-end test,
- prepare database state before executing an assertion.

They are not intended as a model for the final test campaign structure.

They deliberately keep setup, execution, and assertions inside a single test case and do not apply the modular organization expected for the task. The final suite should improve on these examples by using fixtures, helpers, parametrization, clear grouping, and non-duplicated setup where appropriate.

## 2) Coverage Report

### What you must provide

Do not commit generated coverage reports.

The coverage report will be generated automatically by the GitLab pipeline from the submitted test suite.

The tool used for coverage is **`pytest-cov`**, which integrates `pytest` with **`coverage.py`**.

The expected evidence is therefore the output produced by the automated coverage command, not a manually committed report file.

### Expectations

The submitted test suite should:

- run successfully with the project test command,
- allow the pipeline to generate the coverage report,
- produce meaningful coverage on the implementation,
- make uncovered files, lines, or branches visible through the generated tool output.

The generated `pytest-cov` / `coverage.py` output is the official evidence for this task.

## 3) Quality Criteria

### Test quality

The test campaign must show:

- coherent organization,
- clear naming,
- meaningful assertions,
- controlled setup,
- limited duplication,
- appropriate use of fixtures and parametrization,
- sensible separation between unit, integration, and API-level tests when applicable.

### Coverage quality

The coverage effort should prioritize real behavior.

Prefer tests that verify important logic and error handling over tests that only execute simple accessors, trivial constructors, or framework boilerplate.

Remaining uncovered code should be limited to code that is genuinely hard to exercise, environment-specific, defensive, or not relevant to the implemented behavior.

### Maintainability

The resulting tests should be suitable for future development.

They should help detect regressions when the implementation changes, and they should be understandable by another developer without requiring a separate explanatory document.
